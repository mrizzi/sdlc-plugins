## Root Cause

The convention heading extraction in the plan-feature skill uses `line[3:]` to extract the section name after the `## ` prefix. This slicing does not strip trailing whitespace. When a `CONVENTIONS.md` heading line contains trailing spaces (e.g., `## Migration Patterns  `), the extracted dictionary key becomes `"Migration Patterns  "` (with trailing spaces). The subsequent convention lookup performs an exact-match comparison against the normalized name `"Migration Patterns"`, which fails silently -- no warning is logged and the convention is simply omitted from the generated task description.

## Affected Files

- `plugins/sdlc-workflow/skills/plan-feature/SKILL.md` -- convention conformance analysis section
  - Heading extraction: `section_name = line[3:]` (missing `.strip()`)
  - Task enrichment: `if convention_name in discovered_conventions:` (exact match against unstripped key)

## Suggested Approach

Apply `.strip()` to the extracted heading text so that trailing (and leading) whitespace is normalized before storing in the conventions dictionary:

```python
section_name = line[3:].strip()
```

This ensures headings with trailing whitespace match correctly during the lookup phase. No changes are needed to the task enrichment lookup itself -- the fix is entirely in the extraction step.

## Reproducer Strategy

1. Create a test `CONVENTIONS.md` fixture with trailing whitespace on a heading line (e.g., `## Migration Patterns  ` with two trailing spaces).
2. Run the convention extraction logic against this fixture.
3. Assert that the convention is correctly extracted and matched, producing the expected `Per CONVENTIONS.md ...` annotation in the task's Implementation Notes.
4. Verify that a heading without trailing whitespace continues to work (regression check).
