# Criterion 6: Endpoint returns 404 for non-existent SBOM IDs (existing behavior preserved)

## Verdict: PASS

## Analysis

The task requires that the existing 404 behavior for non-existent SBOM IDs is preserved. This is not a new feature but a regression check -- the PR should not break the existing error handling.

The relevant code in the handler is unchanged:

```rust
let sbom = SbomService::new(&db)
    .fetch(sbom_id.id)
    .await
    .context("Failed to fetch SBOM")?;
```

The `SbomService::fetch()` call is present in both the pre-PR and post-PR versions of the handler. When the SBOM ID does not exist, `fetch()` returns an error that is propagated via the `?` operator, ultimately resulting in a response generated by `AppError`'s `IntoResponse` implementation. Based on the repository conventions documented in `common/src/error.rs`, `AppError` maps not-found conditions to HTTP 404.

The PR changes do not modify:
- The `SbomService::fetch()` call or its error handling
- The `AppError` error type or its response mapping
- The order of operations (SBOM lookup still happens before threshold filtering)

The threshold filtering logic is only reached AFTER the SBOM is successfully fetched. If the SBOM does not exist, the handler returns an error before reaching the filtering code. This means the 404 behavior is fully preserved.

The missing test is not evaluated under this criterion (which concerns the behavior itself) but is relevant to the overall test coverage assessment.
