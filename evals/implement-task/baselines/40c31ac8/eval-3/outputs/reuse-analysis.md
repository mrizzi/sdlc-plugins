# Reuse Analysis: TC-9203 -- Add package license filter to list endpoint

## Overview

The task description identifies three Reuse Candidates. All three are used directly in the implementation -- no new utility functions are created that duplicate existing functionality.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:** Handles comma-separated multi-value query parameter parsing and SQL IN clause generation. Given a column reference and a filter string like `"MIT,Apache-2.0"`, it splits on commas, trims whitespace, and generates a `WHERE column IN ('MIT', 'Apache-2.0')` clause compatible with SeaORM query builders.

**How it is reused:** Called directly from `PackageService::list()` in `modules/fundamental/src/package/service/mod.rs` when the `license` query parameter is present. The license filter string (e.g., `"MIT,Apache-2.0"`) is passed to `apply_filter` along with the `package_license` table's license column reference. This avoids writing any custom comma-parsing or IN clause generation logic.

**Why not create new code:** Creating a separate `parse_license_filter` or `build_license_in_clause` function would duplicate the exact behavior that `apply_filter` already provides. The function is designed to be generic across all filter types -- it is already used by the advisory endpoint for severity filtering and can be reused without modification for license filtering.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:** A structurally identical filter implementation for the `severity` query parameter on the advisory list endpoint. It demonstrates:
- How to add an optional filter field to the endpoint's `Query` struct (e.g., `pub severity: Option<String>`)
- How to extract the filter value from the query parameters in the handler function
- How to pass the filter to the service layer
- How to validate the filter value and return 400 Bad Request for invalid input
- The overall structural pattern: Query struct field -> handler extraction -> service call -> `apply_filter` in database layer

**How it is reused:** The advisory endpoint's `Query` struct pattern is replicated in `modules/fundamental/src/package/endpoints/list.rs`. Specifically:
1. The `license: Option<String>` field is added to the package endpoint's `Query` struct, matching the structure of the advisory endpoint's `severity: Option<String>` field
2. The handler function follows the same extraction and validation pattern
3. The service call follows the same parameter-passing pattern
4. Error handling follows the same `Result<T, AppError>` with `.context()` pattern

This is a structural/pattern reuse -- the advisory endpoint serves as the reference implementation that the package endpoint mirrors. No code is copied; instead, the same architectural pattern is followed to maintain consistency across endpoints.

**Why not invent a new pattern:** The advisory endpoint's filter pattern is an established convention in the codebase. Introducing a different approach (e.g., a middleware-based filter, a custom extractor, or a different query struct shape) would create inconsistency between endpoints and make the codebase harder to maintain.

## Reuse Candidate 3: `entity/src/package_license.rs`

**What it provides:** An existing SeaORM entity that maps the `package_license` database table -- the join table between packages and their declared licenses. It includes:
- The entity struct with column definitions (package_id, license identifier)
- SeaORM `Relation` definitions linking back to the `package` entity
- Column enum variants for type-safe column references in queries

**How it is reused:** Used in `PackageService::list()` (in `modules/fundamental/src/package/service/mod.rs`) to perform the JOIN between the `package` table and the `package_license` table when the license filter is active. Specifically:
1. The entity's relation definitions are used with SeaORM's `.join()` API to create the `JOIN package_license ON package.id = package_license.package_id` clause
2. The entity's column enum is used to reference the license column in the `WHERE` clause generated by `apply_filter`
3. No raw SQL is written -- all join and filter logic uses SeaORM's type-safe query builder through this entity

**Why not write raw SQL:** The `package_license` entity already encodes the table structure, column names, and relationships in a type-safe manner. Writing raw SQL (`JOIN package_license ON ...`) would bypass SeaORM's type checking, miss schema changes caught at compile time, and diverge from the project's convention of using entity-based queries throughout.
