# Reuse Analysis for TC-9203

This document details how each Reuse Candidate listed in the task description is used in the implementation, following the skill's "Reuse first" directive.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:** Handles comma-separated multi-value query parameter parsing and SQL IN clause generation.

**How it is reused:** Called directly in `PackageService::list()` (in `modules/fundamental/src/package/service/mod.rs`) to process the `license` query parameter value. No wrapper, no reimplementation. Specifically:

- When the license value is a single string like `"MIT"`, `apply_filter` generates a simple equality condition (`WHERE license = 'MIT'`).
- When the license value is comma-separated like `"MIT,Apache-2.0"`, `apply_filter` splits on commas and generates a SQL `IN` clause (`WHERE license IN ('MIT', 'Apache-2.0')`).

**What is NOT done (avoided duplication):**
- No custom comma-splitting logic is written in the endpoint handler or service layer.
- No manual SQL `IN` clause construction is written.
- No new utility function is created for multi-value parameter parsing.

**Reuse type:** Direct invocation -- the function is called as-is with no modifications to its source.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:** The severity filter implementation is structurally identical to the license filter needed here. It demonstrates the Query struct pattern with an optional filter field.

**How it is reused:** The advisory endpoint's `Query` struct serves as the structural template for the package endpoint's `Query` struct modification. Specifically:

- The advisory `Query` struct has an optional `severity` field (e.g., `severity: Option<String>`) that is extracted from query parameters by Axum's `Query` extractor.
- The package `Query` struct in `modules/fundamental/src/package/endpoints/list.rs` gets a new optional `license: Option<String>` field following the exact same pattern -- same type, same optionality, same serde attributes.
- The handler function in the advisory endpoint passes the severity value to `AdvisoryService::list()`. The package handler follows the same pattern, passing the license value to `PackageService::list()`.

**What is NOT done (avoided duplication):**
- No new query parameter extraction mechanism is invented.
- No new handler pattern is designed -- the advisory handler's control flow (extract query params, pass to service, return paginated result) is mirrored exactly.

**Reuse type:** Structural pattern reuse -- the advisory code is used as a template, and the same architecture is applied to the package module with domain-specific names substituted (severity becomes license, advisory becomes package).

## Reuse Candidate 3: `entity/src/package_license.rs`

**What it provides:** The existing SeaORM entity for the package-license join table, mapping packages to their declared licenses.

**How it is reused:** Used in `PackageService::list()` to construct the JOIN between the `package` table and the `package_license` table when filtering by license. The entity provides:

- The SeaORM `Entity`, `Column`, and `Relation` definitions needed to build the JOIN condition.
- Column references (e.g., `package_license::Column::LicenseId` or equivalent) used in the filter's WHERE clause generated by `apply_filter`.
- The relation definition that connects `package_license` back to `package`, enabling SeaORM to generate the correct JOIN SQL.

**What is NOT done (avoided duplication):**
- No raw SQL JOIN is written -- the SeaORM entity's relation definitions handle JOIN generation.
- No new entity or model is created for the package-license relationship.
- No duplicate mapping logic is written to associate packages with licenses.

**Reuse type:** Direct usage -- the entity is referenced in SeaORM query builder calls with no modifications to its source file.
