# Reuse Analysis: TC-9203 -- Add package license filter to list endpoint

This document analyzes all three Reuse Candidates identified in the task description and
describes how each would be used in the implementation.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:** The `apply_filter` function handles comma-separated multi-value
query parameter parsing and SQL `IN` clause generation. It accepts a raw query parameter
string (e.g., `"MIT,Apache-2.0"`), splits it by comma, and produces a filter condition
suitable for use in a SeaORM query. It handles both single-value (`"MIT"`) and
multi-value (`"MIT,Apache-2.0"`) inputs uniformly.

**How it would be used:** In both `modules/fundamental/src/package/endpoints/list.rs`
and `modules/fundamental/src/package/service/mod.rs`, the `license` query parameter
value would be passed directly to `apply_filter` to generate the filter condition. This
avoids writing any custom comma-splitting logic, any custom SQL `IN` clause builder, or
any new utility function that would duplicate what `apply_filter` already does.

**Why reuse over reimplementation:** The function already encodes the project's
established convention for multi-value query parameter parsing. Writing a new function
to split comma-separated values and build filter conditions would duplicate this
existing, tested utility. All multi-value filter parameters across the codebase should
use `apply_filter` for consistency.

**Integration point:** Called from the list handler (or service method) after extracting
the `license` query parameter from the request. The returned filter condition is applied
to the SeaORM query builder before execution.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:** The advisory list endpoint implements a `severity` query parameter
filter using a pattern that is structurally identical to the license filter needed for
TC-9203. It demonstrates:
- A Query struct with an optional filter field (`severity: Option<String>`)
- Extraction of the filter value from the Axum query parameters
- Delegation to `apply_filter` for parsing and condition generation
- Conditional application of the filter (skip when `None`, apply when `Some`)
- Proper error handling for invalid values via `AppError`

**How it would be used:** The advisory list endpoint serves as the structural template
for the package list endpoint changes. The implementation would follow the exact same
pattern:
1. Add `license: Option<String>` to the package endpoint's Query struct (mirroring
   `severity: Option<String>` in the advisory Query struct)
2. In the handler function, check if `license` is `Some`, and if so, call `apply_filter`
   with the value (mirroring how the advisory handler processes `severity`)
3. Pass the filter condition to the service layer (mirroring how `AdvisoryService`
   receives filter parameters)
4. Handle validation errors the same way the advisory endpoint does

**Why reuse over reimplementation:** Following this proven pattern ensures consistency
across list endpoints in the `modules/fundamental` module. Deviating from the
established pattern would create an inconsistency that future maintainers would need to
reconcile. The advisory endpoint has already validated this approach in production.

**Integration point:** The pattern is followed structurally -- not called as a function.
The advisory endpoint's code is the reference implementation that the package endpoint
mirrors.

## Reuse Candidate 3: `entity/src/package_license.rs`

**What it provides:** The existing SeaORM entity definition for the `package_license`
join table, which maps packages to their declared licenses. This entity includes the
column definitions, relation definitions, and model struct needed to query the
package-license relationship using SeaORM's type-safe query builder.

**How it would be used:** When the license filter is active, the query in
`PackageService::list` needs to JOIN the `package` table with the `package_license`
table to filter by license. Instead of writing raw SQL (`JOIN package_license ON ...`),
the implementation would use the SeaORM entity's relation definitions to construct the
JOIN:
- Use the `package_license::Entity` to reference the join table in the query
- Use the `package_license::Column::License` (or equivalent column name) to apply the
  `IN` clause filter condition generated by `apply_filter`
- Use SeaORM's `.join()` or `.find_also_related()` methods with the entity's defined
  relations

**Why reuse over reimplementation:** The entity already exists and defines the correct
table structure, column names, and relations. Writing raw SQL would bypass SeaORM's
type safety and would not benefit from the entity's relation definitions. Additionally,
if the table schema changes in a future migration, the SeaORM entity will be updated in
one place, and all queries using it -- including this new filter -- will be updated
automatically at compile time.

**Integration point:** Used in the SeaORM query builder within
`modules/fundamental/src/package/service/mod.rs` to construct the JOIN and apply the
filter condition on the license column.

## Summary

| Reuse Candidate | Role in Implementation | Reuse Type |
|---|---|---|
| `common/src/db/query.rs::apply_filter` | Parse comma-separated license values and generate SQL IN clause | Direct function call |
| `modules/fundamental/src/advisory/endpoints/list.rs` | Structural template for Query struct, handler flow, and filter application | Pattern replication |
| `entity/src/package_license.rs` | SeaORM entity for JOIN query between packages and licenses | Entity reference in query builder |

No new utility functions are needed. All comma-separated parsing and filter generation
is handled by the existing `apply_filter` function. The implementation follows the
established advisory endpoint pattern and uses the existing package_license entity for
type-safe JOINs.
