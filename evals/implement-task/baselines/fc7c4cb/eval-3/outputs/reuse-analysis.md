# Reuse Analysis: TC-9203 - Add package license filter to list endpoint

This document analyzes all three Reuse Candidates listed in the task description and explains how each is applied in the implementation.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:** The `apply_filter` function handles comma-separated multi-value query parameter parsing and SQL `IN` clause generation. Given a string like `"MIT,Apache-2.0"`, it splits on commas, produces individual values, and generates the appropriate SQL filter (e.g., `WHERE license_id IN ('MIT', 'Apache-2.0')`).

**How it is reused:** This function is called directly from `modules/fundamental/src/package/service/mod.rs` when the optional `license` parameter is present. The service passes the raw query parameter string to `apply_filter`, which handles all parsing and clause generation. No new utility function is created for this purpose -- `apply_filter` already provides exactly the functionality needed.

**Why reuse is appropriate:** Writing a custom comma-split parser or building SQL IN clauses manually would duplicate the existing logic in `apply_filter`. The function is a shared query helper specifically designed for this use case, and it is already used by other modules (e.g., the advisory module's severity filter). Reusing it ensures consistent behavior across all filter parameters in the API.

**What is NOT done:** No new parsing or filtering utility functions are created. No manual string splitting logic is written. No raw SQL WHERE clause construction is done outside of `apply_filter`.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:** The advisory list endpoint implements a `severity` query parameter using the same structural pattern needed for the `license` filter. It demonstrates:
- How to define an optional filter field on the Query parameter struct
- How to extract the filter value from the deserialized query parameters
- How to pass the filter value to the corresponding service method
- The overall handler function structure (extract params, call service, return paginated response)

**How it is reused:** This file is used as a structural guide (reference implementation) for modifying `modules/fundamental/src/package/endpoints/list.rs`. The package list endpoint handler is modified to follow the same pattern:
1. Add `license: Option<String>` to the package endpoint's Query struct, mirroring how `severity: Option<String>` is defined in the advisory endpoint's Query struct.
2. Extract the license value and pass it to `PackageService::list()`, mirroring how the advisory endpoint extracts the severity value and passes it to `AdvisoryService::list()`.
3. Maintain the same handler return type and error handling patterns (`Result<T, AppError>` with `.context()` wrapping).

**Why reuse is appropriate:** The advisory endpoint's severity filter is structurally identical to the license filter. Following the same pattern ensures consistency across the API's list endpoints, makes the codebase easier to maintain, and reduces the risk of introducing structural inconsistencies. The convention conformance analysis in Step 4 of the skill would confirm this pattern is established across the `endpoints/` directory.

**What is NOT done:** The advisory endpoint code is not copied wholesale -- only its structural pattern is followed. No changes are made to the advisory endpoint itself.

## Reuse Candidate 3: `entity/src/package_license.rs`

**What it provides:** The `package_license` entity is an existing SeaORM entity that maps the join table between packages and licenses. It provides:
- The SeaORM `Entity`, `Model`, `Column`, and `Relation` definitions for the `package_license` table
- Column definitions (e.g., `package_id`, `license_id`) usable in SeaORM query builders
- Relation definitions that connect packages to their licenses, enabling SeaORM joins

**How it is reused:** In `modules/fundamental/src/package/service/mod.rs`, the license filter query joins the `package` table with the `package_license` table using SeaORM's join API and the relations/columns defined in `entity/src/package_license.rs`. Specifically:
1. The service builds a SeaORM `Select` query starting from the `package` entity.
2. It adds a JOIN to `package_license` using the relation defined in the entity.
3. It applies the filter (generated by `apply_filter`) on the `package_license::Column::LicenseId` column.
4. This produces a query equivalent to: `SELECT package.* FROM package JOIN package_license ON package.id = package_license.package_id WHERE package_license.license_id IN ('MIT', 'Apache-2.0')`.

**Why reuse is appropriate:** The `package_license` entity already exists and provides the SeaORM-idiomatic way to reference the join table. Writing raw SQL joins or creating a duplicate entity definition would bypass SeaORM's type safety and violate the project's convention of using SeaORM entities for all database access. The entity is part of the `entity/` crate specifically so that multiple modules can share it.

**What is NOT done:** No raw SQL is written for the join. No new entity or model is created for the package-license relationship. The existing entity is used as-is without modification.
