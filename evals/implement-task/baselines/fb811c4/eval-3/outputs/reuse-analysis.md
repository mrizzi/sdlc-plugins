# Reuse Analysis: TC-9203 — Add package license filter to list endpoint

## Overview

The task description lists three Reuse Candidates. All three are directly applicable and
should be reused as described below. No additional reusable code needs to be discovered
beyond what the task already identifies — the listed candidates cover parsing, filtering,
query construction, and entity relationships comprehensively.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**Source**: `common/src/db/query.rs`

**What it provides**: The `apply_filter` function handles comma-separated multi-value query
parameter parsing and SQL IN clause generation. It accepts a raw query string value (e.g.,
`"MIT,Apache-2.0"`), splits it on commas, trims whitespace, and generates the appropriate
SQL `WHERE column IN (...)` clause for the ORM query.

**How it will be reused**: Called directly in `PackageService::list()` when the `license`
parameter is `Some`. The function is invoked as:

```rust
apply_filter(query, "license", license_value)
```

This eliminates the need to write any custom comma-parsing or IN clause generation logic.
The function is a shared utility already used by other modules (e.g., the advisory severity
filter), so reusing it ensures consistency and avoids duplication.

**Reuse type**: Direct invocation — no modification to `apply_filter` itself is needed. The
function's existing signature and behavior match the license filter requirements exactly.

**Benefit**: Avoids reimplementing comma-separated value parsing and SQL IN clause construction.
Ensures consistent filtering behavior across all list endpoints that support multi-value
parameters.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**Source**: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides**: The advisory list endpoint implements a `severity` query parameter filter
using a structural pattern that is identical to what the license filter requires:

1. A `Query` struct with an `Option<String>` field for the filter parameter
2. Axum's `Query` extractor to deserialize the query string into the struct
3. Extraction of the filter value and passing it to the service layer
4. Validation of the filter value (returning 400 for invalid input)

**How it will be reused**: The advisory handler's pattern is used as a structural template
(not copied verbatim) for modifying the package list handler. Specifically:

- **In `modules/fundamental/src/package/endpoints/list.rs`**: Add a `license: Option<String>`
  field to the existing `Query` struct, mirroring how the advisory handler's `Query` struct
  has a `severity: Option<String>` field
- **In the handler function**: Extract `query.license` and pass it to
  `PackageService::list()`, following the same control flow the advisory handler uses for
  `query.severity`
- **Validation**: Apply the same validation pattern — if the value is present but empty or
  malformed after comma-splitting, return 400 via `AppError`

**Reuse type**: Structural pattern reuse — the advisory handler code is the template that
guides the implementation in the package handler. The advisory code itself is not modified.

**Benefit**: Ensures the license filter follows an established, reviewed pattern already in
production. Reduces implementation risk because the advisory severity filter has been tested
and validated. Makes the codebase consistent — all list endpoints that support filtering use
the same Query struct + service parameter + apply_filter pattern.

## Reuse Candidate 3: `entity/src/package_license.rs`

**Source**: `entity/src/package_license.rs`

**What it provides**: The existing SeaORM entity definition for the package-license join table.
This entity maps packages to their declared licenses, providing the ORM model needed to
JOIN from the `package` table to the `package_license` table when filtering by license.

**How it will be reused**: Used in `PackageService::list()` to construct the JOIN query when
the license filter is active. Instead of writing raw SQL, the implementation uses SeaORM's
relation model:

```rust
// In PackageService::list(), when license filter is active:
// Join Package -> PackageLicense to filter by license SPDX identifier
query = query.join(JoinType::InnerJoin, package_license::Relation::Package.def().rev());
```

The entity's column definitions (e.g., `package_license::Column::License`) are used in the
WHERE clause generated by `apply_filter`.

**Reuse type**: Direct usage of the existing entity — no modification to
`entity/src/package_license.rs` is needed. The entity already defines the table structure,
columns, and relations required for the JOIN.

**Benefit**: Avoids writing raw SQL for the join. Leverages SeaORM's type-safe query building.
Ensures the query uses the same table and column names defined in the entity, preventing
typos or schema drift. The entity is already maintained as part of the codebase's migration
and schema management.
