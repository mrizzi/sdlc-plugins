# Reuse Analysis: TC-9203 -- Add package license filter to list endpoint

This document details how each Reuse Candidate identified in the task description
is used in the implementation. All three candidates are reused; no new utility
functions or parsing logic are created.

---

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:**
`apply_filter` is a shared query-builder helper that accepts a raw query parameter
string (potentially comma-separated) and produces the corresponding SQL filter clause.
For a single value like `"MIT"`, it generates an equality condition. For a
comma-separated value like `"MIT,Apache-2.0"`, it splits on commas, trims whitespace,
and generates a SQL `IN (...)` clause. This function is already used across the
codebase for multi-value filtering.

**How it is reused:**
The `license` query parameter is passed directly to `apply_filter` in the
`PackageService::list()` method. This handles all parsing and SQL generation for
both single-value and multi-value license filters. No custom comma-splitting logic,
no manual `IN` clause construction, and no new utility functions are written. The
function is called exactly as it is called in other filter implementations across
the codebase.

**Why not write new logic:**
Writing a new parser for comma-separated values would duplicate `apply_filter`'s
existing functionality. The function already handles edge cases (whitespace trimming,
empty segments) consistently across all endpoints that use it. Reusing it ensures
the license filter behaves identically to every other multi-value filter in the API.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:**
The advisory list endpoint (`GET /api/v2/advisory`) already implements a `severity`
query parameter using the same filtering approach needed for the license filter. Its
structure includes:
- A `Query` struct with an optional filter field (`severity: Option<String>`)
- A handler function that extracts the Query, passes the filter to the service layer,
  and returns `PaginatedResults<AdvisorySummary>`
- Integration with `apply_filter` for the actual SQL generation

**How it is reused:**
This endpoint serves as the structural blueprint for the package license filter
implementation. The following elements are replicated in
`modules/fundamental/src/package/endpoints/list.rs`:

1. **Query struct pattern:** An optional `license: Option<String>` field is added to
   the package endpoint's Query struct, mirroring how `severity: Option<String>` is
   defined in the advisory Query struct.
2. **Handler flow:** The handler extracts the query parameters, passes the `license`
   value to `PackageService::list()`, and returns the paginated result -- the same
   flow used in the advisory handler for `severity`.
3. **Service integration:** The way the advisory handler passes `severity` to
   `AdvisoryService::list()` and how that service conditionally applies the filter
   is the exact pattern followed in `PackageService::list()` for the `license` filter.

This is structural reuse (following an established pattern) rather than code copying.
The advisory endpoint is the reference implementation that defines the conventions
for adding filter parameters to list endpoints in this codebase.

## Reuse Candidate 3: `entity/src/package_license.rs`

**What it provides:**
The `package_license` SeaORM entity defines the database mapping for the join table
between packages and their declared licenses. It includes:
- The `Entity` struct with table name mapping
- The `Model` struct with column definitions (package_id, license identifier, etc.)
- `Relation` definitions that link to the `package` entity and the license reference
- `Column` enum for type-safe column references in queries

**How it is reused:**
In `PackageService::list()` (within `modules/fundamental/src/package/service/mod.rs`),
when the `license` filter is present, the query builder uses the `package_license`
entity to construct a JOIN:

1. The SeaORM query starts from the `package` entity.
2. A `.join()` call references `entity::package_license::Entity` and its relation
   to `package`, using the entity's built-in relation definitions.
3. The filter condition (generated by `apply_filter`) is applied against
   `entity::package_license::Column::LicenseIdentifier` (or the equivalent column
   name defined in the entity).

This approach uses the existing entity's relation and column definitions rather than
writing raw SQL JOIN clauses. It keeps the query type-safe and consistent with how
other service methods in the codebase construct JOINs via SeaORM entities.

**Why not write raw SQL:**
The `package_license` entity already encodes the table name, column names, and
foreign key relationships. Writing raw SQL would bypass these definitions, creating
a maintenance risk if the schema changes. Using the entity ensures the query is
validated at compile time by SeaORM's type system.

---

## Summary

| Reuse Candidate | Reuse Type | What Is Avoided |
|---|---|---|
| `apply_filter` | Direct function call | Custom comma-parsing logic, manual SQL IN clause generation |
| Advisory list endpoint | Structural pattern (blueprint) | Inventing a new filter parameter pattern, inconsistent handler flow |
| `package_license` entity | SeaORM entity for JOIN | Raw SQL joins, hardcoded table/column names, compile-time safety loss |

No new utility functions are created. No parsing logic is duplicated. All filtering,
query construction, and entity relationships leverage existing code.
