## Architecture Summary

The implementation replaces the `advisory_status` lookup table with a PostgreSQL `advisory_status_enum` enum column directly on the `advisory` table. The change flows through four layers:

1. **Database layer** (Task 2): A single reversible migration creates the enum type, adds the column, backfills data, drops the FK, and drops the lookup table.
2. **Entity layer** (Task 3): The SeaORM entity definition is updated to use the enum column, and the lookup table entity is removed.
3. **Service/API layer** (Task 4): Advisory queries, models, and endpoints are updated to read/filter status from the enum column directly, eliminating the join overhead.
4. **Ingestion layer** (Task 5): The advisory ingestion pipeline writes enum values directly instead of inserting into the lookup table.
5. **Test layer** (Task 6): Integration tests are updated to work with the new schema.
