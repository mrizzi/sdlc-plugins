### Architecture Summary

This feature replaces the `advisory_status` lookup table (joined via `status_id` FK) with a PostgreSQL `advisory_status_enum` column directly on the `advisory` table. The change touches four layers:

1. **Database** — A reversible migration creates the enum type, adds and backfills the column, then drops the FK and lookup table.
2. **Entity** — SeaORM entity definitions are updated to use `DeriveActiveEnum` for the new enum field; the `advisory_status` entity is removed.
3. **Service/Endpoints** — All advisory queries in `AdvisoryService` are updated to filter on the enum column directly, eliminating the join. Endpoint handlers pass through the enum value as a string (no API response change).
4. **Ingestion** — The advisory ingestion pipeline writes enum values directly instead of inserting into the lookup table first.

All changes are delivered via feature branch `TC-9005` because the migration and code changes are mutually dependent — merging either side alone would break the application.
